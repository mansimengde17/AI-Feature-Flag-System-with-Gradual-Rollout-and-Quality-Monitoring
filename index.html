Feature flags that understand quality

AI Feature Flag System with Gradual Rollout

Feature flags built for AI features, where working is a quality gradient and not a boolean. Flags roll out in stages, quality is scored against the baseline continuously, and a bad variant rolls back on its own.

Why this system exists

The operational problem it solves, and why teams keep rebuilding it by hand until someone gives them the tooling.

Why boolean flags fall short

A traditional flag tells you the new code path did not crash. It cannot tell you the new prompt quietly made summaries worse for a fifth of your users. AI features fail on a gradient, so the flag has to measure quality, not just on or off.

Why it lands interviews

It proves you understand that shipping AI features needs different operational patterns than shipping normal code, and that you can build the tooling to support that difference.

What it protects against

A degraded prompt reaching everyone. Rollout is staged, each stage is gated on quality, and a sustained drop in the worst decile triggers an automatic rollback with an alert.

Why the numbers are trustworthy

Users are assigned by a consistent hash, so raising the percentage only adds users and never reshuffles them, and a canary t test blocks any advance that is statistically worse than baseline.

The exact workflow

Every request or task travels this path. The strip lights up in sequence so you can see the order the stages fire in.

01
Evaluate

The SDK hashes the user id into a stable bucket and returns baseline or experimental.

02
Serve

The app renders the chosen variant; integration is three lines and never blocks on the flag service.

03
Score

Every output is queued for asynchronous judge scoring so quality measurement adds no user latency.

04
Window

Scores flow into rolling windows tracking mean, worst decile, and trend per variant.

05
Gate

At each stage boundary a canary test compares experimental to baseline before advancing.

06
Rollback

A sustained breach of the quality floor sets the rollout to zero and fires a Slack alert.

How it works in depth

The mechanisms behind each stage, and the design decisions that make the system behave well under real load.

1
Quality aware flag schema

Beyond a percentage, each flag carries a quality threshold, a rollback trigger, a staged schedule, targeting rules, and both the baseline and experimental variants as first class fields.

2
Consistent assignment

A hash of flag name and user id maps to a stable bucket, so the same user always gets the same variant and raising the rollout percentage is purely additive.

3
Asynchronous quality monitor

After every gated response a judge score is queued, never blocking the user. Rolling windows track mean, standard deviation, worst decile, and whether the trend is improving, stable, or degrading.

4
Staged rollout with canary gate

The schedule steps one, five, twenty five, fifty, one hundred percent. Each boundary runs a Welch t test against baseline; the rollout advances only when the variant is statistically no worse.

5
Automatic rollback

When the worst decile stays below the floor for a sustained window the rollout drops to zero, an alert fires, and a cooldown prevents flapping back and forth.

6
Drop in SDK

Three lines integrate the client, which caches flag config locally and falls back to baseline if the service is unreachable. Shadow mode runs the variant with zero user impact to catch catastrophic failures first.

How to run it

Runs fully offline with deterministic providers, so the demo and the tests reproduce byte for byte. Add API keys to switch to live models.

# offline: both rollout scenarios end to end
pip install -r requirements.txt
python demo.py
# -> good variant advances to 100%, bad variant auto rolls back

# serve the flag service
uvicorn aiflags.api:app --app-dir src --port 8000
curl -s -X POST localhost:8000/v1/flags/subject-gen-v2/evaluate \
  -d '{"user_id": "u-42"}'
